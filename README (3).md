# Claude Design Prompt — Wren Law Firm Redesign

**How to use this file.** Open Claude Design at claude.ai/design. Attach the three project files (`README.md`, `URL_INVENTORY.md`, `SOURCE_OF_TRUTH.md`). Paste the prompt below into the chat. Claude Design will read the attachments, build a complete bilingual mockup with your brand constraints respected, and prepare a handoff bundle ready for Claude Code → GitHub.

## The prompt

I'm handing off the full redesign of **Wren Law Firm**, a 29-year-old personal injury, Social Security disability, and workers' compensation practice in Little Rock, Arkansas. I'm giving you full creative control within the constraints below.

I've attached three files that together are the complete source of truth for this project:

- **`README.md`** — the current design system audit (brand tokens, components, layout rules, what's working, what needs fixing)
- **`URL_INVENTORY.md`** — every URL on the current site, what each page is for, and which design template it uses
- **`SOURCE_OF_TRUTH.md`** — the factual content foundation: firm facts, attorney bios, verified credentials, voice, ready-to-use copy blocks, and the bilingual (English + Spanish) template scope

**Read all three files before designing anything.** They reconcile — `SOURCE_OF_TRUTH.md` is canonical for content, `README.md` is canonical for brand tokens, `URL_INVENTORY.md` is canonical for page structure.

### What I want

A complete, modern, prospect-facing website design system and template set for Wren Law Firm, delivered as a bilingual (English + Spanish) mockup with a Claude Code handoff bundle ready to push to GitHub.

This is a **credibility-first law firm site**, not a volume-ad PI firm. A worried Arkansan who was just hurt in a car accident needs to feel, in the first five seconds, that this firm is experienced, direct, and reachable right now. That's the job the design has to do.

### Creative direction

You have full creative control over visual language, layout, motion, and interaction — **within these constraints**:

1. **Brand palette is fixed.** Navy `#19283f` and gold `#b69d74` are the primary colors. The navy + gold pairing is the firm's identity and is non-negotiable. You can introduce supporting neutrals freely.
2. **Typography pairing is the starting point.** Playfair Display (serif display) + Open Sans (body) is the current system. Feel free to propose a refreshed pairing if you have strong conviction — just justify the change with a single sentence in your response so the firm can weigh it.
3. **Logo stays.** The existing courthouse/pediment logo is strong and identifies the firm. Use it as-is in the header and footer. Commission an SVG version in your output if you're rebuilding the asset pipeline.
4. **Feel: conservative but modern.** Traditional law-firm signifiers (columns, seals, serif headlines) are appropriate. What the current site lacks is modern polish — generous whitespace, confident typography, crisp photography treatment, meaningful motion. Bring that in.
5. **Avoid generic AI-design tropes.** No purple-blue gradients, no floating glass cards, no "generated by AI" Midjourney-stock photography. This is a law firm from 1997 in Arkansas.

### Handoff bundle requirements — "ready for GitHub"

When the design is approved, package it into a Claude Code handoff bundle that produces a **production-grade repo** ready to push to GitHub. The repo must be implementation-agnostic enough that the firm's developer can choose their own stack, but opinionated enough to get 80% of the way there without further direction.

The handoff bundle should specify:

1. **Recommended stack.** Next.js 14+ with App Router, TypeScript, and Tailwind CSS is the default recommendation. If you choose differently, justify it in the handoff README. Whatever stack you pick:
   - Server-rendered pages (not SPA-only) for SEO.
   - `next-intl` or equivalent for the English/Spanish routing with `hreflang` tags on every page.
   - Static generation where possible; ISR for the blog.
   - Image optimization built-in.

2. **Repo structure.**
   ```
   /app
     /(en)/...
     /(es)/...
     /api/contact-form
   /components
     /ui/           ← primitives
     /sections/     ← page-level composed components
     /forms/
   /content
     /en/           ← MDX or markdown for static content
     /es/
     /blog/
   /lib
     /design-tokens.ts   ← CSS custom properties exported as typed tokens
     /schema.ts          ← JSON-LD LegalService structured data
     /analytics.ts
   /public
     /logo.svg
     /attorneys/
     /og/
   /styles
     /globals.css
   README.md
   CLAUDE.md          ← project context for future Claude Code sessions
   .env.example
   package.json
   ```

3. **Design tokens file** (`lib/design-tokens.ts`) with the canonical set from §4 of `README.md` — every color, every type ramp, every spacing value, every breakpoint. Use CSS custom properties (`:root { --color-navy: #19283f; ... }`) and export matching TypeScript constants. This replaces the current site's scattered per-page CSS.

4. **`CLAUDE.md` at repo root.** This is context for future Claude Code sessions. Include: project overview, which files are canonical for what, where content lives, how to add a new blog post, how to add a new practice area, how translations work, what's in the design token file and how to modify it.

5. **Structured data (JSON-LD) component** that renders `LegalService` schema on every page with the firm's real address, phone, `areaServed: "Arkansas"`, attorney subtype, and `inLanguage: "en"` or `"es"` per variant.

6. **Form handling.** Contact and consultation request forms should POST to an API route that the developer can wire to Gravity Forms, Resend, Formspree, or the firm's CRM of choice. Scaffold the route with a clear TODO for the integration.

7. **Accessibility baseline.** Visible focus rings, semantic landmarks, proper heading hierarchy, labeled form fields (not placeholder-only — this is a current-site defect called out in the README), alt text on all images, keyboard-navigable mobile menu, WCAG AA contrast.

8. **Performance baseline.** Subset fonts (Playfair 300, 400, italic; Open Sans 400, italic — not all 18 weights the current site loads). Preconnect for font CDN. Lazy-load below-the-fold images. Avoid render-blocking scripts.

9. **An `.env.example`** with placeholders for Google Tag Manager, Meta Pixel (if retained), reCAPTCHA site key, and the contact-form provider. No real values.

10. **A deployment-ready `README.md` at repo root** explaining: what this is, how to install, how to run dev, how to add content, how to deploy to Vercel / Netlify / the firm's host of choice, how to swap the form provider, how to add or edit translations.

## Notes on using this prompt

- **Attach all three files** — don't paste their contents into the prompt. Claude Design reads attachments natively and the prompt becomes unreadable if you inline them.
- **Don't over-specify visual direction up front.** The prompt is deliberately light on "make the hero look like X" — that's what Claude Design's iteration loop is for. Over-specifying at this stage wastes the tool's strength.
- **Ask for the creative direction statement first** (step 1 of "what I want back"). This catches misreads cheaply. If the statement lands wrong, redirect before any pixels are drawn.
- **Iterate on the homepage before approving anything else.** Once the homepage design system is locked, everything else extends from it. Don't let Claude Design race ahead to template 5 while the hero still isn't right.
- **Use Claude Design's inline comments and sliders** during iteration rather than long text descriptions. That's what they're built for.
